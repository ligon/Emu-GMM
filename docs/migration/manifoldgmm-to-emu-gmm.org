#+TITLE: Porting Consumers from =manifoldgmm.GMM= to =emu_gmm.estimate=
#+AUTHOR: emu-gmm
#+OPTIONS: toc:2

This note documents the port pattern from
[[file:../../../ManifoldGMM][ManifoldGMM]]'s class-based driver
=manifoldgmm.GMM(restriction=...).estimate(...)= to Emu-GMM's functional
=emu_gmm.estimate(...)=. The trigger was the Seasonality call-site
inventory tracked in [[https://github.com/ligon/Emu-GMM/issues/1][Emu-GMM
issue #1]]; the same pattern applies to any consumer that today
constructs a =GMM= driver around a =MomentRestriction=.

* Mapping

| ManifoldGMM (v1)                                    | Emu-GMM                                                                         |
|-----------------------------------------------------+---------------------------------------------------------------------------------|
| =MomentRestriction(g, X)=                           | =EmpiricalMeasure(x=X, mask=..., weights=...)= (or =EmpiricalMeasure.from_pandas=) |
| =GMM(restriction=r)=                                | (no driver class --- bundle parameters at the =estimate= call site)             |
| =g(theta, X)= per-observation residual              | =psi(x, theta)= per-observation residual (argument order swapped)               |
| =restriction.clusters=                              | =ClusteredCovariance(cluster_ids=...)=                                          |
| =gmm.estimate(weighting='two_step')=                | =weighting=ContinuouslyUpdated()= (CU subsumes two-step in v1)                  |
| =gmm.estimate(weighting='cu')=                      | =weighting=ContinuouslyUpdated()= (default)                                     |
| =res.theta_array=                                   | =result.theta_hat= (preserves the user's parameter dataclass type)              |
| =res.criterion_value=                               | =result.J_stat=                                                                 |
| =res.cov_matrix=                                    | =result.Sigma_theta= (labelled =haliax.NamedArray=)                             |
| Manifold-aware =Positive(1, 1)= parameter           | Flat \(\mathbb{R}^K\) (Riemannian manifold for params deferred to v2)           |

** Argument-order convention

ManifoldGMM's =moment_func= signature is =g(theta, X)=; Emu-GMM's
=StructuralModel= signature is =psi(x, theta)=, where =x= is a *single*
observation row, not the whole sample. The implications are:

1. *Order is swapped.* You must call your residual with the observation
   first, parameters second.
2. *=psi= is per-observation.* The framework handles =vmap= /
   integration internally; your residual sees one row at a time.

The signature change is intentional --- it's the v1 commitment to
=Measure= as the integration operator (see =docs/design.org= Section
2). Consumers that today batch over =X= inside =g= must hoist the
per-observation kernel into a =psi(x, theta)= form.

** Missing-data convention (the NaN-mask story)

Seasonality's Euler scripts (and most pandas-backed call sites) carry
missingness as NaN inside =X=. ManifoldGMM's =MomentRestriction= uses
=nansum= / =np.where(isnan, 0, .)= internally. Emu-GMM mirrors the
boundary intent (NaN is a natural sentinel at the I/O edge) but takes a
*per-cell* view of missingness inside the framework, not a row-level
one. The motivating case --- a per-asset return that is observed only
for households that hold the asset --- is intrinsically per-cell: row
=i= can have asset =a= observed and asset =b= missing, and the
estimator must compute a separate per-asset \(N_j\) rather than dropping
the entire row.

*** =from_pandas= and the =nan_aware= switch

=EmpiricalMeasure.from_pandas= takes a keyword-only =nan_aware=
parameter (default =True=). With =nan_aware=True= and no explicit
=mask=:

- The per-coordinate mask is inferred from =~df.isna()= *element-wise*,
  giving an =(N, M)= 0/1 matrix (not a row-level =~df.isna().any(axis=1)=).
- NaN cells in the stored =x= array are replaced with zero so they
  cannot poison downstream JAX arithmetic when a model =psi= happens to
  read a masked-out cell.

Set =nan_aware=False= to preserve a legacy "all-ones mask, NaN
verbatim" behaviour. An explicit =mask= argument always takes precedence
over NaN inference.

For raw-array inputs, =EmpiricalMeasure.from_nan_aware(x, weights=...)=
is the array-side convenience constructor with the same per-cell
semantics.

*** The "double where" pattern (and why single-where is wrong)

The hot path in =expectation= and =jacobian= uses two complementary
=jnp.where= guards:

1. *Pre-sanitise =x=*: before invoking the user's =psi=, replace
   =NaN= cells in =x= with zero
   (=x_safe = jnp.where(jnp.isnan(x), 0.0, x)=). This means the user's
   residual never sees =NaN= inputs --- even at masked-out cells where
   the residual is undefined --- so neither the primal nor the JVP/VJP
   tape carries =NaN=.
2. *Mask the residual output*: after =psi= produces a per-row =(M,)=
   vector (or the gradient a =(M, K)= block), zero out cells where
   =mask == 0= /before/ multiplying by the per-coordinate weight
   =d_ij * w_i=. The contraction never multiplies a real weight by a
   non-finite output.

Only one of these guards is *not* enough:

- Sanitising =x= alone leaves a =psi= that legitimately returns =NaN= at
  unobserved cells (a natural pattern, e.g., =log(return)= for a
  non-holder) free to poison the moment sum via =0 * NaN = NaN=.
- Masking the output alone --- writing
  =jnp.where(mask, psi(x), 0.0)= without sanitising =x= --- looks
  safe in the primal but fails under reverse-mode AD. Cotangent flow
  through =jnp.where= visits *both* branches: the cotangent for the
  =psi(x)= branch is computed even where =mask == 0= and is then masked
  to zero, but only after the VJP of =psi= has already been evaluated
  at the =NaN= input. The resulting cotangent contains =NaN=, the mask
  step computes =0 * NaN = NaN=, and the gradient of the CU criterion
  is poisoned.

This is the same trap discussed in the JAX FAQ entry on gradients
through =where=: the safe pattern is to ensure the branch that the
gradient evaluates at sees only finite values. The double-where pattern
is the implementation of that rule for the NaN-mask case.

*** Boundary behaviour summary

- =EmpiricalMeasure.from_pandas(df)= with =nan_aware=True= (default)
  derives a per-cell mask from =~df.isna()= and replaces NaN cells in
  =x= with zero. The mask is the authoritative source of
  observability; downstream computations never see NaN inputs.
- =EmpiricalMeasure.from_nan_aware(x, weights=...)= is the equivalent
  for raw arrays.
- If you construct =EmpiricalMeasure= directly with =mask= and =x=,
  NaN entries at *masked-out* positions are still safely dropped from
  the moment estimator, the Jacobian, and the covariance --- the
  double-where guard in the hot path takes care of it. NaN at
  *observed* (=mask = 1=) positions propagates --- that case is a
  residual-evaluation bug, not a missingness signal.

restriction = MomentRestriction(g=g, X=X)

** After (Emu-GMM)

#+BEGIN_SRC python
import jax.numpy as jnp
import jax_dataclasses as jdc
from emu_gmm import (
    EmpiricalMeasure,
    IIDCovariance,
    ContinuouslyUpdated,
    estimate,
)

@jdc.pytree_dataclass
class Theta:
    theta: jnp.ndarray  # length-M parameter vector

def psi(x, t):
    """Per-observation residual: returns (M,) vector."""
    return x[:-1] - t.theta * x[-1]

# Boundary conversion: per-cell NaN-aware mask from a pandas DataFrame.
# nan_aware=True is the default; pass nan_aware=False to opt out.
measure = EmpiricalMeasure.from_pandas(df)         # df: shape (N, D)
# Or, with explicit JAX arrays:
#   measure = EmpiricalMeasure(
#       x=x_jax, mask=mask_jax, weights=jnp.ones(N),
#   )
# Or, raw-array NaN-as-missing:
#   measure = EmpiricalMeasure.from_nan_aware(x_jax)

result = estimate(
    model=psi,
    measure=measure,
    covariance=IIDCovariance(),                   # or ClusteredCovariance(...)
    weighting=ContinuouslyUpdated(),              # default; CU subsumes two-step
    theta_init=Theta(theta=jnp.zeros(M)),
)

theta_hat = result.theta_hat.theta
J_stat = result.J_stat                            # already a Python float
#+END_SRC

* Cluster-robust path

ManifoldGMM keeps =cluster_ids= on the =MomentRestriction= and the
driver dispatches automatically. In Emu-GMM clustering is a separate
strategy on the same =EmpiricalMeasure=:

#+BEGIN_SRC python
from emu_gmm import ClusteredCovariance

result = estimate(
    model=psi,
    measure=measure,
    covariance=ClusteredCovariance(
        cluster_ids=cluster_ids_jax,
        n_clusters=int(cluster_ids_jax.max()) + 1,
    ),
    theta_init=Theta(theta=jnp.zeros(M)),
)
#+END_SRC

The =Measure= / =CovarianceStrategy= split is one of the v1
architectural commitments (=CLAUDE.md=, point 1): the same
=EmpiricalMeasure= pairs with =IIDCovariance= or =ClusteredCovariance=
without changes elsewhere in the pipeline. Both covariance
implementations apply the same double-where guard so that =NaN= cells
at masked-out positions cannot poison =V_X=.

* Weighting policy

| ManifoldGMM call           | Emu-GMM equivalent             |
|----------------------------+--------------------------------|
| =weighting='two_step'=     | =ContinuouslyUpdated()=        |
| =weighting='cu'=           | =ContinuouslyUpdated()=        |
| =weighting='identity'=     | =Identity()=                   |
| =weighting='fixed', W=...= | =Fixed(V0=...)= (note: pass V0 not W; the framework's whitening uses \(V^{-1/2}\) internally) |

CU and two-step are *asymptotically equivalent* but not identical in
finite samples (=CLAUDE.md=, point 4). Emu-GMM v1 makes CU the
default; iterated weighting (the literal two-step path) is deferred to
v2 because CU captures the same asymptotic behaviour with a smoother
finite-sample objective.

* Result-field translation table

| ManifoldGMM =GMMResult=        | Emu-GMM =EstimationResult=                          |
|--------------------------------+-----------------------------------------------------|
| =result.theta_array=           | =result.theta_hat= (preserves dataclass)            |
| =result.criterion_value=       | =result.J_stat= (Python float)                      |
| =result.optimizer_report=      | =result.diagnostics.optimizer_info= (dataclass)     |
| =result.cov_matrix=            | =result.Sigma_theta= (=haliax.NamedArray=)          |
| =result.moment_cov=            | =result.V_X= (=haliax.NamedArray=)                  |
| (computed downstream)          | =result.J_dof=, =result.J_pvalue= (computed inline) |

For DataFrame-ready output: =result.to_pandas()= materialises the
labelled fields.

* What this port does *not* cover (yet, in v1)

These ManifoldGMM features have no v1 Emu-GMM equivalent and are
deferred to v2+ (see =CLAUDE.md= "Deferred to v2+"):

- =Positive(1, 1)= and other Riemannian parameter spaces. Use a flat
  reparameterisation (e.g. =log_sigma= rather than =sigma=) in v1.
- =LoggingTrustRegions= optimiser. Use =optimistix_lm()= (default) or
  =scipy_lm()=.
- =MomentWildBootstrap= and similar inference helpers; bootstrap loops
  must be written by the caller in v1.
- =KStatisticResult=, =ridge_inverse=, and other identification helpers.
- =StratifiedCovariance=, =ReplicateWeightCovariance=.
- True iterated weighting (subsumed by CU for v1).

If your call site exercises one of these features, the migration is
*not* mechanical --- coordinate with the maintainer before porting.
