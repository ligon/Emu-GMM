#+DATE: 2026-05-29
#+AUTHOR: Synthesis subagent

* Executive summary

Four lenses (A: orthogonality, B: commitments, C: coherence, D: design-drift)
audited the post-wf6 codebase against ~docs/design.org~, ~docs/api-sketch.org~,
~docs/mcar-asymptotics.org~, and ~CLAUDE.md~. After 16 PRs (wf4 #43-#46 plus
wf5 #30-#38, #46-#49) the architectural core is *intact*: all eight CLAUDE.md
commitments were probed at runtime and held; the orthogonal-interface algebra
of ~design.org~ §2 is preserved; the Cholesky pipeline and G'ΛG information
construction are honored; float64 is enabled before any sub-module import; no
silent math errors were found.

The drift is *documentation-side and surface-side*, not algorithmic:

1. =docs/design.org= sections §3 (Euler walkthrough still single-asset), §5
   (no mention of #47 diagonal-V dispatch, #45 Welch-Satterthwaite scheme),
   §6 (NamedArray-bool mask aspirational; ~coef_table~'s pandas-direct egress
   undocumented) have not been refreshed to match six substantive landings
   (#31, #34, #35, #45, #47, #49) plus the multi-asset Euler example.
2. =docs/migration/manifoldgmm-to-emu-gmm.org= lists four features (MomentWildBootstrap,
   KStatisticResult, ridge_inverse, true iterated weighting) as deferred-to-v2+
   when PRs #32, #33, #34, #36 all landed before wf6. The ~two_step~ weighting
   row also still maps to ~ContinuouslyUpdated()~.
3. The public-surface result-type regime is split three ways
   (mutable / frozen-leaf / frozen-pytree) with no documented invariant.
   ~ClusterBootstrapResult~ is a frozen leaf while its three sibling
   inference results (~JTestResult~, ~KStatisticResult~, ~WildBootstrapResult~)
   are pytrees. vmap over the four inference helpers succeeds for three of
   them and fails for the fourth.
4. ~IteratedWeighting~ (#34) satisfies the ~WeightingStrategy~ protocol
   on the surface but the iterated outer loop is implemented as an
   ~isinstance(weighting, IteratedWeighting)~ branch in ~estimator.py~ —
   the one place the orthogonal algebra is type-tag-dispatched rather than
   protocol-dispatched.
5. ~LabelContext~ is the declared type of ~EstimationResult.labels~ but
   lives in ~_internal/~ and is not re-exported.

Overall verdict: *minor-cleanups*. No redesign required; the implementation
is correct and the architectural commitments are preserved. The fixes are a
documentation sweep, a small set of surface cleanups, and one polymorphism
lift (~IteratedWeighting~ → ~requires_outer_loop~ hook on ~WeightingStrategy~).

* Cross-lens convergent findings (high confidence)

A finding that surfaces in two or more lenses has higher confidence — it is
not lens-specific blind-spotting. Five clusters of convergence emerged.

** XL-1. ~IteratedWeighting~ is an isinstance branch, not a protocol method

   *Lens convergence*: A (probe 4), D (§4 narrative caveat that the iterated
   /algorithm/ lives in ~estimator.py:113-275~, not in ~weighting.py~).

   *Severity*: medium.

   ~estimator.py~ line 520 has ~isinstance(weighting, IteratedWeighting)~ which
   dispatches into a Python-level outer-loop driver (~_run_iterated_weighting~,
   lines 113-275) instead of routing through the
   ~WeightingStrategy.whitening_residual~ protocol. The strategy nominally
   satisfies the protocol via a CU-fallback ~whitening_residual~ (~weighting.py~
   lines 297-316), but the actual iterated control flow is type-tagged in the
   estimator.

   Lens D notes that this is architecturally clean in one sense — the inner
   Fixed-weight residual correctly does not see ∇θV_k, so the per-iteration
   gradient is what design.org §4 wants — but Lens A is right that the lift
   from "this is the way iterated weighting works" to "this is a hook in the
   ~WeightingStrategy~ protocol" was not done. The current state is the only
   place a strategy /instance/ rather than protocol /method/ drives estimator
   control flow.

   *Recommended action*: lift ~_run_iterated_weighting~ behaviour into a
   protocol method (e.g. ~requires_outer_loop: bool~ + ~outer_loop_driver()~
   hook). Estimator dispatches polymorphically. File as issue, fix as ~v1.1~.

** XL-2. Migration guide is stale on four landed features

   *Lens convergence*: C (F-C1, high severity), D ("the implementation outpaces
   the docs").

   *Severity*: high.

   ~docs/migration/manifoldgmm-to-emu-gmm.org~ lines 221-261 still tell the
   user that ~MomentWildBootstrap~, ~KStatisticResult~, ~ridge_inverse~,
   and "true iterated weighting" are deferred to v2+. PRs #32 (wild bootstrap),
   #33 (ridge_inverse), #34 (IteratedWeighting), #36 (K-statistic) all merged
   before wf6. The weighting-policy table also still maps ~weighting='two_step'~
   to ~ContinuouslyUpdated()~ which is no longer the correct port now that
   ~IteratedWeighting~ exists.

   A user reading the guide before reaching for the package will not realise
   the inference machinery they need is already in the box. This is the
   highest-leverage doc fix.

   *Recommended action*: doc-only PR (no code changes); update the four
   bullets in "What this port does NOT cover" and the ~two_step~ row in the
   weighting table.

** XL-3. design.org §3-§6 trails the implementation on six landings

   *Lens convergence*: D (primary; six itemised drifts), C (notes that "the
   API surface holds together but design.org has not been refreshed"),
   indirectly B (no commitment violation but each drift is the documented
   half of a commitment).

   *Severity*: medium (collective).

   Six places in ~design.org~ are stale:

   1. §3 specifies single-asset M=1 Hansen-Singleton; implementation
      (~examples/euler.py~) is multi-asset M=3, J_dof=1.
   2. §3 says "log-normal preference shock simulator parameterised at theta";
      implementation's ~euler_sampler_factory.sampler~ ignores theta (common
      random numbers path).
   3. §5 specifies V* = V + τ·diag(V); implementation (#47) adds a diagonal-V
      dispatch V* = V + τ·(tr V / M)·I to restore κ-control on diagonal V.
   4. §5 specifies "regularization-adjusted critical values"; implementation
      (#45) reports ~J_pvalue_adjusted~ via Welch-Satterthwaite moment-matching
      (not Davies / Imhof). Named in mcar-asymptotics.org Theorem 6 but not
      in design.org.
   5. §6 specifies ~mask ∈ NamedArray[Obs, Moments, bool]~; implementation
      stores ~mask~ as plain ~Float[Array, "N M"]~ (float 0/1) for jit.
   6. §6's "labelled outputs route via NamedArray" claim is weakened by
      ~coef_table~ (added by #44) which goes flat-params → pandas directly,
      bypassing the intermediate NamedArray step.

   The implementation is correct; design.org has been outpaced. Each
   instance is one paragraph of doc update.

   *Recommended action*: single doc-only PR sweeping design.org §§3-6 with
   the six clarifications above.

** XL-4. Result-type regime is split three ways

   *Lens convergence*: C (F-C2, high severity), A (probe 6 notes
   ~OptimizerInfo~ is now a pytree but says nothing about siblings).

   *Severity*: high.

   Eight result-like dataclasses across the surface:

   | Type                     | Decorator                       | Pytree? | Mutable? |
   |--------------------------+---------------------------------+---------+----------|
   | ~EstimationResult~       | ~@dataclass~                    | leaf    | yes      |
   | ~Diagnostics~            | ~@dataclass(frozen=True)~       | leaf    | no       |
   | ~OptimizerInfo~          | ~@jdc.pytree_dataclass~         | pytree  | no       |
   | ~JTestResult~            | ~@jdc.pytree_dataclass~         | pytree  | no       |
   | ~KStatisticResult~       | ~@jdc.pytree_dataclass~         | pytree  | no       |
   | ~WildBootstrapResult~    | ~@jdc.pytree_dataclass~         | pytree  | no       |
   | ~ClusterBootstrapResult~ | ~@dataclass(frozen=True)~       | leaf    | no       |
   | ~LabelContext~           | ~@dataclass(frozen=True)~       | leaf    | no       |

   Two practical consequences:

   1. ~vmap(j_test)~ over a panel of nulls works; the equivalent for
      ~vmap(cluster_bootstrap)~ over a panel of seeds fails at the output
      (frozen leaf, not pytree). ~ClusterBootstrapResult~ is the outlier
      among the four inference results.
   2. ~Diagnostics~ contains ~OptimizerInfo~ and is itself a leaf, so PR #46's
      promotion of ~OptimizerInfo~ to a pytree only buys pytree behaviour in
      isolation — wrapped in ~Diagnostics~ it is immediately re-buried.

   *Recommended action*: decide invariant ("inference results are pytrees;
   estimation results are leaves") and lift ~ClusterBootstrapResult~ to
   ~@jdc.pytree_dataclass~. Document the invariant in a one-paragraph note
   in api-sketch.org. Filed as ~v1.1~ fix.

** XL-5. ~LabelContext~ is private but typed on a public field

   *Lens convergence*: C (F-C3, medium severity), B notes the LabelContext
   route is preserved but does not flag the leakage.

   *Severity*: medium.

   ~EstimationResult.labels~ has declared type ~LabelContext~ (types.py:359);
   ~LabelContext~ lives in ~src/emu_gmm/_internal/labels.py~ and is not in
   ~__all__~. A user typing ~help(result.labels)~ or ~result.labels.__class__~
   lands in ~emu_gmm._internal.labels~, which CLAUDE.md treats as off-limits.

   *Recommended action*: re-export ~LabelContext~ from ~emu_gmm.types~ (or top
   level) and add to ~__all__~. One-line fix.

* Lens-specific concerns (blind-spot indicators)

Findings that surfaced in only one lens — these are either lens-specific
nits or blind-spots that the other three lenses missed.

** A-only

*** A-1. Cached intermediates tuple is undocumented contract

    PR #49's cached tuple ~(m, psi_safe, weight_mask, N_j)~ between
    ~EmpiricalMeasure.expectation_and_contributions~ and
    ~IID/ClusteredCovariance~ is undocumented at the protocol level. Lens A
    rates this low (no break today, only ~AnalyticalCovariance~ does not
    consume it, third-party measure works via fallback). Worth noting: this
    is the seed of a contract that will calcify as more strategies adopt
    the pattern. Lens A's recommendation (lift to a typed intermediate
    dataclass when the population exceeds 2 covariance types) is sound.

*** A-2. ~apply_fixed_tau~ algebraic fallback assumes canonical Tikhonov form

    Anchor-tau hoist (#45) falls back to ~V + tau_anchor * diag(V)~ for
    third-party regularisers lacking ~apply_fixed_tau~. This assumes the
    canonical form; third-party regularisers using different ridge bases
    (e.g. ~EigenvalueFloor~, ~NearestPSD~ — listed in CLAUDE.md "Deferred to
    v2+") would diverge from their own ~apply()~. Not a bug today (no such
    regularisers exist), but a contract trap that will be hit when the
    regulariser zoo expands.

** B-only

    No findings. Lens B's contribution is positive verification: each of
    the eight CLAUDE.md commitments was probed with a concrete runtime
    experiment whose expected outcome would diverge if the commitment were
    violated; all eight passed. Notably:

    - Commitment 3 (anchor-once-then-freeze) was forced into a regime where
      re-anchoring would give τ=1000 while the anchored τ=0; the framework
      correctly used the frozen 0.
    - Commitment 4 (CU's gradient retains V-dependence) was measured: the
      jax.grad of CU half-objective differs from a frozen-V naive variant
      by (-0.08, -0.25) on a test problem — V-trace preserved through
      Cholesky + forward_solve.
    - Commitment 7 (float64 before sub-module imports) was confirmed via
      instrumented load: zero ~emu_gmm.*~ submodules import before
      ~jax.config.update("jax_enable_x64", True)~.

*** C-1. Naming inconsistencies (F-C5, F-C6, F-C7, F-C8)

      in a second "Inference helpers" group lower down. No distinction.

    ~build_diagnostics~, ~build_optimizer_health~, ~compute_cond_info~ sit at
    the top level despite being construction helpers used only by
    ~estimator.py~ and two unit tests (the tests import from the submodule
    directly). Putting them on ~__all__~ adds three kwargs-heavy functions to
    the user surface that nobody needs. Lens C rates F-C4 medium.

** Issue: Lift IteratedWeighting outer loop into WeightingStrategy protocol

   *Severity*: medium. *Lens*: A (probe 4), D (§4 caveat).

   #+BEGIN_QUOTE
   The post-#34 ~IteratedWeighting~ strategy satisfies the ~WeightingStrategy~
   protocol via a CU-fallback ~whitening_residual~, but the actual iterated
   algorithm is implemented as ~_run_iterated_weighting~ in
   ~src/emu_gmm/estimator.py:113-275~ and dispatched via
   ~isinstance(weighting, IteratedWeighting)~ at estimator.py:520. This is
   the one place a strategy /instance/ rather than protocol /method/ drives
   estimator control flow.

   This is *correct* in the sense that the inner Fixed-weight residual's
   gradient correctly does not see ∇θV_k (which design.org §4 requires for
   iterated), but the orthogonal-interface algebra in design.org §2 is
   leaked — three other strategies (Identity, Fixed, ContinuouslyUpdated)
   dispatch through the standard residual_fn → optimizer path, only
   IteratedWeighting through a special one.

   *Proposal*: lift the outer-loop behaviour into a protocol hook. Options:

   1. Add ~requires_outer_loop: bool~ + ~outer_loop_driver(model, measure,
      ...) → (theta, info)~ optional methods to ~WeightingStrategy~. Estimator
      dispatches polymorphically: ~if weighting.requires_outer_loop:
      theta, info = weighting.outer_loop_driver(...)~.
   2. Move the iterated driver into ~IteratedWeighting.estimate(...)~ as a
      bound method and let the estimator delegate.

   Acceptance criteria:
   - No isinstance branches in ~estimator.py~ against weighting strategies.
   - All existing IteratedWeighting tests pass unchanged.
   - A third-party weighting strategy needing an outer loop can plug in
     without touching estimator.py.
   #+END_QUOTE

** Issue: Unify result-type pytree regime

   *Severity*: high. *Lens*: C (F-C2).

   #+BEGIN_QUOTE
   Eight result-like dataclasses across the public surface split three ways:

   - mutable leaf: ~EstimationResult~
   - frozen leaf: ~Diagnostics~, ~ClusterBootstrapResult~, ~LabelContext~
   - frozen pytree: ~OptimizerInfo~, ~JTestResult~, ~KStatisticResult~,
     ~WildBootstrapResult~

   Two consequences:

   1. ~vmap(j_test)~ / ~vmap(k_statistic)~ / ~vmap(moment_wild_bootstrap)~
      work; ~vmap(cluster_bootstrap)~ fails at the output. The outlier is
      ~ClusterBootstrapResult~.
   2. ~OptimizerInfo~ was promoted to a pytree in PR #46 specifically so the
      ~(theta_opt, info)~ tuple returned by ~Optimizer~ survives jit. But
      ~Diagnostics~ contains ~OptimizerInfo~ and is itself a leaf, so the
      user-visible path through ~result.diagnostics.optimizer_info~ re-buries
      the pytree inside a leaf.

   *Proposal*: declare and document the invariant. Recommended:
   inference results are pytrees (already the case for three of four);
   estimation / diagnostics are leaves (already the case). Lift
   ~ClusterBootstrapResult~ to ~@jdc.pytree_dataclass~. Document the
   invariant in api-sketch.org and add to a "Public-surface conventions"
   subsection of CLAUDE.md or design.org §6.
   #+END_QUOTE
