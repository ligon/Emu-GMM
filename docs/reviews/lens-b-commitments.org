* Commitment 3 — Pairwise overlap → adaptive Tikhonov; anchor-once-then-freeze
PRESERVED.

Probe (post-#45, post-#47): construct a problem where
~cond(V(theta_init)) = 22~ (below ~kappa_target=100~, so tau anchors at 0)
but ~cond(V(theta_hat)) ≈ 9200~ (would re-anchor at tau=1000 if the policy
were violated). ~result.diagnostics.tau_realised == 0~, matching the
anchor at ~theta_init~ and *not* re-anchoring at ~theta_hat~. Confirmed
that ~estimator.py~ lines 405-419 call ~regularization.apply(V0)~ once at
~theta_init~ then route the residual closure through ~_apply_anchored~,
which delegates to ~regularization.apply_fixed_tau(V, tau_anchor)~ for
DiagonalTikhonov (line 417-419) and the algebraic form
~V + tau_anchor * diag(V)~ for arbitrary RegularizationStrategy.

Post-#47 diagonal-V branch: separately verified that ~apply_fixed_tau~
correctly dispatches the diagonal branch
(~V + tau * (tr(V)/M) * I~) but still uses the *anchor* tau, not a re-anchored
one. Probe: anchor at diagonal ~V0 = diag(1, 1, 1e-4)~ to get
~tau0 ≈ 0.015~; apply at ~V_hat = diag(1, 1, 1e-2)~ where a re-anchor
would give ~tau = 0~. ~apply_fixed_tau~ returns the diagonal-branch
algebraic form at the *frozen* tau0, not 0.

Verified at: ~src/emu_gmm/regularization.py~ lines 220-257 (~apply_fixed_tau~
takes an externally-supplied ~tau~ and dispatches diagonal vs canonical on
the supplied V's structure, but never re-runs the bisection).

* Commitment 4 — CU gradient retains ∇θV term
PRESERVED.

Probe: construct a tiny problem where
- ~m(theta) = (theta.a - 1, theta.b - 2)~
- ~V(theta) = diag(theta.a^2 + 1, theta.b^2 + 1)~

Compute ~jax.grad~ of ~0.5 * ||L(theta)^{-1} m(theta)||^2~ under
~ContinuouslyUpdated.whitening_residual~ vs the "naive" version that freezes
V at the current theta and only varies m. The two gradients differ
(~(-0.48, -0.75)~ vs ~(-0.40, -0.50)~), confirming AD threads through the
Cholesky and the V dependence. The IteratedWeighting addition (#34) did
not perturb the CU class — its ~whitening_residual~ still recomputes the
Cholesky inside the call (weighting.py:215-217), preserving the V trace.

* Commitment 5 — Sigma_theta via G' Λ G, never numerical Hessian
PRESERVED.

Verified at: ~src/emu_gmm/estimator.py~ lines 620-634 inside the post-#45
hoisted ~_compute_inference~ jit'd helper:
#+begin_src python
L_local = cho.cholesky(V_star_local)
...
G_local = jnp.asarray(G_local_raw)  # from measure.jacobian
Z_local = jax.scipy.linalg.solve_triangular(L_local, G_local, lower=True)

Also verified at: ~src/emu_gmm/diagnostics.py~ lines 290-296 inside
~compute_cond_info~ (#31): the same ~Z = solve_triangular(L, G); info = Z.T @ Z~

the half-objective for inference purposes.

The penalty Hessian (#35, when penalty != None) is added via

itself uses only the data G' Λ G — the penalty Hessian flows only into the
~'raw'~ view of ~cond_info~.

Notes:
- Commitment 3 is the one most at risk under future refactors because the
  anchor-once-then-freeze policy is split across two files (the anchor is
  computed in ~estimator.py~ and applied via ~_apply_anchored~, which calls
  ~regularization.apply_fixed_tau~ when available). The probe explicitly
  forced a regime where the realised tau differed from a re-anchored tau,
  and the framework correctly used the realised (frozen) value. Post-#47's
  diagonal-V dispatch is also exercised through ~apply_fixed_tau~ — both
  branches respect the freeze.
- Commitment 5 has a subtle interaction with the penalty hook (#35): the
  penalty Hessian ~H_p~ is added in only ~cond_info['raw']~, never into
  Sigma_theta itself. Sigma_theta remains the data-only ~(G' Λ G)^{-1}~,
  matching the delta-method-variance interpretation.
- Commitment 4: IteratedWeighting (#34) follows a different path
  (Fixed-weight inner loop) but does *not* alter the CU pathway —
  ~ContinuouslyUpdated.whitening_residual~ still recomputes the Cholesky
  inside the call, so AD picks up the V dependence as before. The new
  CU-fallback in ~IteratedWeighting.whitening_residual~ also recomputes
  the Cholesky, but that fallback is only used outside the iterated
  driver.
