#+TITLE: Lens A — Interface Orthogonality Audit (post-wf6)
#+DATE: 2026-05-29
#+AUTHOR: Lens A reviewer

* Scope

Audit of the wf4/wf5 merges (PRs #30–#49) against the orthogonal-interface
algebra committed to in =docs/design.org= §2:

- =StructuralModel= : observation-level :math:`\psi`, :math:`\nabla_\theta\psi`.
- =Measure= : :math:`\mathbb{E}_\mu[\psi]`, :math:`\mathbb{E}_\mu[\nabla_\theta\psi]`.
- =CovarianceStrategy= : :math:`V_\mu(\theta)`.
- =WeightingStrategy= : :math:`\Lambda_\mu(\theta)`.
- =RegularizationStrategy= : :math:`V \to V^\star`.
- =Optimizer= : NLLS solver.
- =Estimator= : composes the above.

Default posture: "violation found"; refute only when verified.

* Probes

** Probe 1 — Cached intermediates (#49): does it leak Measure internals into CovarianceStrategy?

*Question*: =IID/ClusteredCovariance.covariance= and =SyntheticCovariance.covariance=
now accept a =cached_intermediates= kwarg. Does this leak Measure internals
(=psi_safe=, =weight_mask=, =N_j=) into the =CovarianceStrategy= contract?

*Code locations*:
- =src/emu_gmm/covariance/iid.py= lines 71–129
- =src/emu_gmm/covariance/clustered.py= lines 77–138
- =src/emu_gmm/covariance/synthetic.py= lines 55–105
- =src/emu_gmm/covariance/analytical.py= lines 58–73 (does NOT accept the kwarg)
- =src/emu_gmm/estimator.py= lines 436–448 (signature probe + fallthrough)
- =src/emu_gmm/types.py= lines 102–113 (protocol contract — unchanged)
- =src/emu_gmm/measures/empirical.py= lines 164–243 (=expectation_and_contributions=)
- =src/emu_gmm/measures/synthetic.py= lines 68–107 (=moments_and_contributions=)

*Verdict*: PRESERVED. Rationale:

1. =types.CovarianceStrategy= protocol (lines 102–113) is unchanged: it
   still advertises only =covariance(psi, theta, measure) -> (M, M)=.
   =api-sketch.org= §"CovarianceStrategy (Protocol)" lines 200–206 still
   matches.
2. =AnalyticalCovariance.covariance= deliberately does *not* take
   =cached_intermediates= and is the proof that the kwarg is *optional*.
3. The estimator gates the fast path with =inspect.signature= introspection
   (estimator.py lines 436–448): if the user supplied a covariance whose
   =covariance= method lacks =cached_intermediates=, =_cache_method= is set
   to =None= and the call site routes through the back-compat
   =measure.expectation= + =covariance.covariance(model, theta, measure)=
   path (estimator.py lines 479–481, 615–617).
4. Verified empirically: a hand-rolled "third-party" =CovarianceStrategy=
   with the minimal =def covariance(self, psi, theta, measure)= signature
   runs through =estimate()= against an =EmpiricalMeasure= and converges
   to the same =theta_hat= as =IIDCovariance=. See probe transcript in
   Section "Reproductions" below.

The kwarg is what the design's "computational interface" language allows
for: an optimisation hint that bypasses the protocol but does not redefine
it. The cached tuple is generated by the measure's own helper
(=expectation_and_contributions= /
=moments_and_contributions=); the covariance treats it as opaque, the
estimator wires it. Third-party implementations of either side remain
viable.

*Minor concern (low)*: the cached tuple shape =(m, psi_safe, weight_mask, N_j)=
is an undocumented quasi-contract between =EmpiricalMeasure= and
=IID/ClusteredCovariance=. A user writing a third-party empirical-style
measure that exposes =expectation_and_contributions= would have to reproduce
the exact 4-tuple. Mitigation: doc-string on =EmpiricalMeasure.expectation_and_contributions=
spells it out (lines 164–211). Recommendation: when this proliferates
beyond the current 2 covariance types, lift the shape into a typed
intermediate dataclass on =_internal=.

** Probe 2 — Measure protocol extension: =jacobian_contributions= (#36)

*Question*: =EmpiricalMeasure= and =SyntheticMeasure= grew a
=jacobian_contributions= method. Is this an organic extension or an ad-hoc
addition to satisfy the K-statistic's needs? Is it on
=AnalyticalMeasure= too? Is the contract documented?

*Code locations*:
- =src/emu_gmm/measures/empirical.py= lines 378–432 (=jacobian_contributions=)
  and lines 275–340 (=moment_contributions=).
- =src/emu_gmm/measures/synthetic.py= lines 139–192.
- =src/emu_gmm/measures/analytical.py= (does NOT implement either).
- =src/emu_gmm/inference/k_statistic.py= lines 194–249 (=_compute_d_tilde=).
- =src/emu_gmm/inference/wild_bootstrap.py= line 299 (uses =moment_contributions=).
- =src/emu_gmm/types.py= lines 86–98 (=Measure= protocol — unchanged).

*Verdict*: PRESERVED. Rationale:

1. =Measure= protocol (types.py lines 86–98) still advertises only
   =expectation= and =jacobian=. The new methods are concrete-class
   extensions, NOT additions to the protocol.
2. =moment_contributions= and =jacobian_contributions= compose
   organically: they expose the per-observation /contribution/ matrices
   that =expectation= and =jacobian= aggregate. They are the building
   blocks any resampling / weak-identification routine needs, not a
   K-statistic-specific bolt-on. The wild-bootstrap helper (=inference/wild_bootstrap.py=
   line 299) consumes the same =moment_contributions= primitive, and the
   raw-data cluster bootstrap and #13 helpers will compose against the
   same surface.
3. =AnalyticalMeasure= deliberately does NOT implement either method:
   there are no "observations" to contribute. K-statistic
   (=k_statistic.py= lines 232–240) duck-types on
   =hasattr(measure, "moment_contributions")=; the analytical fall-through
   sets the strong-identification correction to zero with a documented
   asymptotic equivalence under :math:`m(\theta_0) = 0=. This is the
   correct fallback for the analytical measure under the K-statistic
   semantics.

*Minor concern (low)*: the protocol contract (=types.Measure=) only
advertises =expectation= / =jacobian=. The new =*_contributions= methods
are an implicit "extended Measure" interface used by inference helpers,
but the documentation only mentions them as ManifoldGMM-style building
blocks in module docstrings. Recommendation: either (a) lift them into
the protocol as optional methods with a documented sentinel for
analytical measures, or (b) document the extended interface explicitly in
=api-sketch.org=.

** Probe 3 — Penalty hook (#35): does it stay separate?

*Question*: Did =PenaltyStrategy= stay separate from =WeightingStrategy= /
=RegularizationStrategy= / =StructuralModel=? Does =compute_cond_info=
with =penalty_hessian= leak penalty internals into Diagnostics?

*Code locations*:
- =src/emu_gmm/penalty.py= (new module).
- =src/emu_gmm/estimator.py= lines 287–288 (kwarg), 458–514 (residual
  augmentation), 723–741 (penalty_hessian construction).
- =src/emu_gmm/diagnostics.py= lines 229–325 (=compute_cond_info=).
- =src/emu_gmm/types.py= Diagnostics docstring lines 273–306.

*Verdict*: PRESERVED. Rationale:

1. =PenaltyStrategy= lives in its own module (=penalty.py=) with its own
   protocol. It is not folded into =WeightingStrategy= or
   =RegularizationStrategy=.
2. The penalty enters the residual vector via concatenation of a
   :math:`\sqrt{p(\theta)}` row (estimator.py line 512). This is the
   minimal-invasive coupling: the NLLS surface :math:`\|r\|^2` absorbs
   the penalty, the weighting/regularization layers see only the
   data-side residual, and the LM Jacobian picks up the parameter-space
   contribution via JAX AD. No changes to =WeightingStrategy.whitening_residual=
   or =RegularizationStrategy.apply= signatures.
3. =compute_cond_info= accepts an optional =penalty_hessian: (K, K) array=
   (diagnostics.py lines 229–325). This is the *data type* of the Hessian
   (a plain JAX array), not a =PenaltyStrategy= reference. The Diagnostics
   layer never sees the penalty object itself; it receives the *materialised*
   Hessian, computed by the estimator via =jax.hessian= over the penalty
   evaluated on flat theta (estimator.py lines 738–741). The diagnostics
   layer treats it as a generic (K, K) symmetric matrix.
4. =Diagnostics= dataclass exposes =final_objective_data= /
   =final_objective_full= as the documented split. This is the right
   discipline: the data criterion (the "GMM J-statistic") and the
   "penalised criterion the optimiser saw" are different objects, both
   reported. =cond_info["raw"]= vs =cond_info["data_only"]= mirrors the
   same split for the information matrix.

No isinstance(penalty, ...) branches exist anywhere outside the
=if penalty is None:= guard that controls residual concatenation
(estimator.py lines 461, 723). This is a clean if-supplied switch on
optional functionality, not a protocol violation.

** Probe 4 — IteratedWeighting outer loop (#34)

*Question*: Does =IteratedWeighting= stay inside the
=WeightingStrategy= contract? Or does the estimator have an
=isinstance(IteratedWeighting)= branch that bypasses the protocol?

*Code locations*:
- =src/emu_gmm/weighting.py= lines 220–316.
- =src/emu_gmm/estimator.py= line 520 (isinstance check), lines 113–275
  (=_run_iterated_weighting=).

*Verdict*: CONCERNING-DRIFT (medium). Rationale:

1. =IteratedWeighting= satisfies the =WeightingStrategy= protocol via a
   CU-fallback =whitening_residual= (weighting.py lines 297–316). So the
   protocol contract is honored on the surface.
2. However, =estimator.py= line 520 has =isinstance(weighting, IteratedWeighting)=:
   when this matches, the estimator dispatches into the
   =_run_iterated_weighting= driver instead of the standard
   residual_fn -> optimizer path. The iterated *algorithm* is implemented
   in the estimator, not in the weighting strategy.
3. This is the one place the orthogonal algebra slips: a strategy
   ostensibly satisfying the =WeightingStrategy= protocol triggers a
   completely different control-flow in the estimator. From an external
   "which weighting will I pick?" perspective the four strategies look
   parallel; from the estimator's perspective three of them
   (=Identity=, =Fixed=, =ContinuouslyUpdated=) dispatch through one
   path and =IteratedWeighting= through another.
4. The design's =WeightingStrategy= advertises "Returns y = L^{-1} m where
   V = L L^T" (types.py docstring). =IteratedWeighting= cannot honor that
   contract in a single call because the "weighting" depends on the
   iterate index — a temporal concept the protocol does not expose.
5. CLAUDE.md "Deferred to v2+" explicitly lists "Iterated weighting:
   subsumed by CU for v1; would be ~50 lines." So v1's design statement
   was that iterated lives outside the protocol algebra. PR #34 reversed
   that decision but did not lift the resulting algorithm into the
   strategy contract.

*Why this is "concerning-drift" not "broken"*: third-party
=WeightingStrategy= implementations still work via the standard path;
only =IteratedWeighting= gets special treatment. The =isinstance= check
is centralised in one place. But the abstraction is leaky in the
direction the design specifically warned against.

*Recommended fix*: either lift the iterated outer loop into a generic
"requires_outer_loop" hook on =WeightingStrategy= (making the dispatch
polymorphic rather than type-tagged), or document that
=IteratedWeighting= is an =Optimizer=-adjacent shim that re-uses the
=WeightingStrategy= type-slot for ergonomic reasons rather than for
algebraic ones. The current state — isinstance branch with an
"sometimes-fallback" =whitening_residual= — is a minor decoupling
failure that will tempt future "isinstance" branches.

** Probe 5 — Anchor-tau hoisting (#45)

*Question*: Did moving =regularization.apply()= out of =residual_fn=
require the estimator to know about =RegularizationStrategy= internals
(e.g. =tau_anchor= field, =apply_fixed_tau= method)?

*Code locations*:
- =src/emu_gmm/regularization.py= lines 220–257 (=apply_fixed_tau=).
- =src/emu_gmm/estimator.py= lines 405–419 (anchored ridge hoist).
- =src/emu_gmm/types.py= lines 134–146 (=RegularizationStrategy= protocol).

*Verdict*: PRESERVED. Rationale:

1. =RegularizationStrategy= protocol (types.py lines 134–146) advertises
   only =apply(V) -> (V_star, tau)=. The =apply_fixed_tau= method is a
   =DiagonalTikhonov=-specific extension, not a protocol addition.
2. =estimator.py= lines 410–419 (=_apply_anchored=) uses =hasattr(regularization,
   "apply_fixed_tau")= and falls back to the algebraic form
   =V + tau_anchor * jnp.diag(jnp.diag(V))= for third-party regularisers.
3. Verified empirically: a hand-rolled =ThirdPartyRegularizer= with only
   =apply()= (no =apply_fixed_tau=, no =tau_threshold=) runs through
   =estimate()= and converges to the same =theta_hat= as the default
   =DiagonalTikhonov=. See probe transcript.
4. Similar pattern for =tau_threshold= (estimator.py line 107): uses
   =getattr(regularization, "tau_threshold", None)= with a
   None-fallback. Third-party regularisers don't need to expose it; the
   =binding_ridge= diagnostic flag is set to False.

*Minor concern (low)*: the algebraic fallback assumes the design's
canonical-form regulariser :math:`V^\star = V + \tau \cdot \text{diag}(V)=. A
third-party regulariser that uses a different ridge basis (e.g.
=EigenvalueFloor=, =NearestPSD=) would diverge from its own
=apply()='s anchored tau under the algebraic fallback. Recommendation:
when the regulariser zoo expands (CLAUDE.md "Deferred to v2+" lists
=EigenvalueFloor=, =NearestPSD=), the protocol contract should include
=apply_fixed_tau= as an optional method, with the algebraic form
deprecated to a "lossy fallback" warning.

** Probe 6 — OptimizerInfo as pytree (#46)

*Question*: Did making =OptimizerInfo= a pytree force =Optimizer=
protocol changes that break =scipy_lm= or other backends?

*Code locations*:
- =src/emu_gmm/types.py= lines 169–187 (=OptimizerInfo=).
- =src/emu_gmm/optimizer.py= lines 47–270 (both adapters).

*Verdict*: PRESERVED. Rationale:

1. =OptimizerInfo= is registered as a pytree via
   =@jdc.pytree_dataclass=, with =status= and =backend= as
   =jdc.static_field()=. The =steps= and =final_objective= fields are
   typed =Any= so they can be Python scalars (eager) or 0-d JAX arrays
   (under jit).
2. =optimistix_lm= (optimizer.py lines 60–117): constructs
   =OptimizerInfo(steps=..., final_objective=..., status=...,
   backend="optimistix")= directly. Under jit =steps= and
   =final_objective= are traced JAX scalars; the pytree allows the tuple
   =(theta_opt, info)= to be a valid jit return value.
3. =scipy_lm= (optimizer.py lines 174–230): constructs the same
   =OptimizerInfo= with =steps=int(result.nfev)=,
   =final_objective=float(result.cost)=. These are concrete Python
   scalars but the pytree dataclass accepts them transparently. The
   =Optimizer= protocol's runtime semantics (eager Python) are
   unaffected.
4. The pytree-ification was needed so the optimistix path could
   return =(theta_opt, info)= as a tracable tuple. The scipy path
   doesn't need it but doesn't break under it.

* Summary

| Probe                                  | Verdict           |
|----------------------------------------+-------------------|
| 1: cached_intermediates leak           | preserved (minor) |
| 2: jacobian_contributions on Measure   | preserved (minor) |
| 3: PenaltyStrategy separation          | preserved         |
| 4: IteratedWeighting isinstance branch | concerning-drift  |
| 5: anchor-tau hoist                    | preserved (minor) |
| 6: OptimizerInfo as pytree             | preserved         |

Overall: *minor-drift* with one *concerning-drift* in =IteratedWeighting=
control flow (probe 4). All other PRs respect the orthogonal-interface
algebra; the IteratedWeighting=type-tag dispatch is the only place the
estimator special-cases a strategy instance instead of routing through
the protocol.

The recurring pattern across probes 1, 2, 5 is identical and correct:
the framework advertises a minimal protocol; concrete framework-provided
implementations add optional methods; the estimator detects them via
=inspect.signature= / =hasattr= / =getattr= and falls back to the
protocol-only path for third-party implementations. This is the right
discipline — but it does mean a third-party that wants the
fast/precise/extended behaviour now has an undocumented "extended
contract" to honour. As the optional-extensions accumulate (cached
intermediates, =jacobian_contributions=, =apply_fixed_tau=,
=tau_threshold=, =penalty_hessian= shape), the gap between
"protocol-only" and "framework-quality" implementations widens.
Recommendation: collect the optional extensions into a documented
"v1.x extended Measure / Covariance / Regularizer" appendix in
=api-sketch.org=, so third-party implementers know what to write to
unlock the fast paths.
